#### 1. SoundReadTools.java — 5 个工具全部接入真实 Service

| 工具 | 之前（假数据） | 之后（真实 Service） |
|------|-------------|-------------------|
| `listVoices()` | 硬编码 5 个音色字符串 | 查 `sys_voice` 表，只返回 2.0 情感音色 |
| `generateScript()` | 调 LLM 生成（已是真实） | 保持不变，优化了 prompt |
| `analyzeEmotion()` | 调 LLM 分析（已是真实） | 保持不变，优化了输出格式 |
| `synthesizeSpeech()` | 返回假字符串 | **调真实 TtsV2Service** → WebSocket 火山引擎 → R2 存储 → 返回真实音频URL |
| `listMyWorks()` | 返回硬编码 3 个假作品 | 查 `user_creation` 表，返回真实创作记录 |

关键代码：
```java
// synthesizeSpeech — 真实调用火山引擎 TTS 2.0
TtsV2Request request = new TtsV2Request();
request.setText(text);
request.setVoiceType(voiceId);
request.setUserKey(user.getId().toString());

TtsV2Response response = ttsV2Service.synthesize(request);
String audioUrl = response.getAudioUrl();

// 保存到创作库
UserCreation creation = new UserCreation();
creation.setUserId(user.getId());
creation.setType("workshop");
creation.setAudioUrl(audioUrl);
creationService.save(creation);
```

## 三、技术架构

```
用户输入 "帮我做一段深夜电台独白"
    │
    ▼ POST /api/agent/chat
    │
AgentController
    │ 选模型（minimax → deepseek → qwen）
    ▼
SmartAssistant.chat(message)
    │ LangChain4j AiServices 代理
    │ LLM 读取 @SystemMessage + @Tool 列表
    │ LLM 决策：依次调用 generateScript → synthesizeSpeech
    ▼
SoundReadTools
    │
    ├─ generateScript("深夜独白", "温柔", 200)
    │     └─ LlmRouter.getChatModelWithFallback() → LLM 生成台本
    │
    ├─ synthesizeSpeech(台本, "zh_female_shuangkuaisisi_moon_bigtts")
    │     ├─ TtsV2Service.synthesize() → WebSocket → 火山引擎 Seed-TTS 2.0
    │     ├─ R2StorageAdapter.uploadAudio() → Cloudflare R2 CDN
    │     └─ CreationService.save() → 写入 user_creation 表
    │
    ▼ 返回综合回复（含音频 URL）
    │
AiWorkshop.vue
    ├─ 解析回复 → 渲染消息气泡
    ├─ 提取音频 URL → 渲染播放器
    └─ 识别步骤 → 渲染 Pipeline 可视化
```

## 五、Bug 修复 & 企业级架构升级

### Bug 1：`<think>` 标签泄漏
- **现象**：DeepSeek 模型的内部推理过程（`<think>...</think>`）直接显示给用户
- **修复**：前端加正则过滤 `reply.replace(/<think>[\s\S]*?<\/think>/gi, '').trim()`

### Bug 2：`NotLoginException: prefix=Bearer`
- **现象**：Tool 方法调 `authService.getCurrentUser()` 时 Sa-Token 找不到 token
- **根因（两层）**：
  1. 前端用裸 `axios` 而非项目的 `request` 模块，没加 `Bearer ` 前缀
  2. LangChain4j 内部线程调 Tool 时不在 HTTP Request 上下文，Sa-Token 无法读取 token
- **修复**：
  1. 前端改用 `import request from '../api/request'`（自动注入 Bearer）
  2. 后端用 **ThreadLocal** 桥接：Controller 层获取 User → `SoundReadTools.setCurrentUser(user)` → Tool 读取 → `finally` 清理

### Bug 3：Agent 理解力差（总在问问题）
- **现象**：用户说"合成语音试试"，Agent 反问"你要什么内容？什么音色？"
- **根因（两层）**：
  1. SystemMessage 太温和，没教 Agent "直接干活"
  2. 没有 ChatMemory，每次对话都是全新的，用户说"2"不知道是什么
  3. 前端选了场景但没传给后端
- **修复**：
  1. SystemMessage 改为"少问多做，直接出活"风格
  2. 加 `@MemoryId` + `chatMemoryProvider` 多轮记忆
  3. 前端传 `scene` 字段，后端注入上下文前缀

### 架构升级：ConcurrentHashMap → Caffeine + chatMemoryProvider

**之前（有 OOM 风险）：**
```java
// 每用户一个 Agent 实例，无上限无过期
ConcurrentHashMap<String, SmartAssistant> agentCache  // ❌
```

**之后（企业级）：**
```java
// 全局单 Agent + 按用户隔离 ChatMemory
SmartAssistant agent;  // 全局唯一

// Caffeine 管理 Memory 生命周期
Cache<Object, Object> memoryCache = Caffeine.newBuilder()
    .maximumSize(500)            // 最多 500 并发会话
    .expireAfterAccess(30min)    // 30分钟不活跃回收
    .recordStats()               // 统计监控
    .build();

// LangChain4j 原生多用户模式
ChatMemoryProvider memoryProvider = memoryId ->
    MessageWindowChatMemory.builder()
        .id(memoryId)
        .maxMessages(20)
        .build();
```

新增 `/api/agent/stats` 监控端点（命中率/淘汰数/当前会话数）。

## 七、面试话术

### 快速版（30秒）

> "我在声读项目中做了一个 **AI 声音工坊** 模块。核心是用 LangChain4j Tool Calling 实现了一个智能制作人——用户用自然语言描述需求，Agent 自动编排多个工具完成台本创作、情感分析、智能选角、语音合成的全链路制作流程。比如用户说'帮我录一段深夜电台独白'，Agent 会自动写台本、选一个温柔的女声、调火山引擎 TTS 2.0 合成真实语音。这个模块从 LLM 决策到 TTS 引擎到 CDN 分发，是端到端落地的。"

### 深入版（面试追问时用）

> **Q：Tool Calling 具体怎么实现的？**
>
> "用 LangChain4j 的 `@AiService` 声明式接口。核心是三个部分：
> 1. **工具定义**：`@Tool` 注解标注 Java 方法，描述文字告诉 LLM 什么时候调用
> 2. **Agent 组装**：`AiServices.builder()` 注册 ChatModel + Tools + ChatMemoryProvider
> 3. **LLM 决策**：调用 `chat(memoryId, msg)` 时，LLM 根据 SystemMessage 和工具描述自主决定调哪个工具、传什么参数
>
> 底层是 OpenAI Function Calling 协议，LangChain4j 帮你把 `@Tool` 序列化成 JSON Schema 发给 LLM，收到 `tool_calls` 响应后反射调用 Java 方法。"

> **Q：遇到过什么困难？**
>
> "三个主要问题：
> 1. **无限循环** — Kimi-K2.5 对终止信号支持不好，收到工具结果后再次发 tool_calls。切到 MiniMax/DeepSeek + SystemMessage 明确禁止 + 30 秒超时三层防御解决。
> 2. **认证上下文丢失** — LangChain4j 内部线程调 Tool 时 Sa-Token 找不到 HttpRequest。用 ThreadLocal 在 Controller 层桥接 User 对象给 Tool 方法。
> 3. **内存泄漏风险** — 早期用 ConcurrentHashMap 缓存 Agent 实例，没有上限和过期。改用 Caffeine 缓存（500 上限 + 30min TTL + LRU 淘汰）+ LangChain4j chatMemoryProvider 原生多用户模式。"

> **Q：这个和直接调 API 有什么区别？为什么用 Agent？**
>
> "传统方式是前端选模式 → 后端 if/else 路由 → 调固定 API。用户必须按你设计的 UI 流程走。
> Agent 方式是用户用自然语言描述需求，LLM 自主决策该调哪些工具、以什么顺序、传什么参数。用户说'帮我做一段深夜电台'，Agent 自动编排 generateScript → synthesizeSpeech 两个工具。如果用户说'太短了，词再多一些'，Agent 理解上文重新调工具。这是从**硬编码工作流**到**语义驱动工作流**的跨越。"

>
> "三层：前端意图拦截 → ReAct 思考框架（单次 prompt 指导思维，零延迟）→ 严禁规则画红线。"

> **Q：用户体验怎么优化？**
>
> "分阶段提示解决等待焦虑 + 自动预加载播放消除缓冲感 + 合成后快捷操作支持多轮编辑。"

>
> "多模型兼容（Kimi 无限循环/DeepSeek 泄漏 think 标签）、认证上下文穿透（ThreadLocal 桥接）、默认参数陷阱（voiceId 不存在导致重试链 50s→37s）。"

> **Q：和 if/else 路由的本质区别？**
