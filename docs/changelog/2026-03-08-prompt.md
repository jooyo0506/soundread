## 一、发现的问题

### 1.1 三层提示词冲突

系统 Prompt 由三层拼装而成，但各层字数限制互相矛盾：

```
最终 System Prompt = DB (ai_role) + Java Agent (getOutputFormatRule) + 前端 (promptHint)
```

| Agent | DB ai_role | Java 代码 | 前端 promptHint | 冲突 |
|-------|-----------|----------|----------------|------|
| 🌙 Radio | "600字以内" | "200-350字" | "600字铁律" | ❌ 代码说 350，其他说 600 |
| 📚 Lecture | "600字以内" | "250-400字" | "600字铁律" | ❌ 代码说 400，其他说 600 |
| 🛒 Ad | "600字以内" | "150-250字" | "600字铁律" | ❌ 代码说 250，其他说 600 |
| 🎨 PictureBook | 无硬限 | "80-180字" | "300-600字" | ❌ 严重冲突 |
| 📰 News | 无硬限 | "200-300字" | — | ⚠️ 偏短 |

**后果**：LLM 收到矛盾指令，生成内容长度不稳定，有时过短有时被截断。

### 1.2 maxOutputTokens 物理截断

所有 Agent 的 `maxOutputTokens` 基于旧的短文案设计，远低于 600 字所需的 ~900 tokens：

| Agent | 旧 maxTokens | 能生成约 | 需要 |
|-------|-------------|---------|------|
| 🎨 PictureBook | 500 | ~330 字 | 600 字❌ |
| 🌙 Radio | 700 | ~460 字 | 600 字❌ |
| 🛒 Ad | 600 | ~400 字 | 600 字❌ |
| 📚 Lecture | 900 | ~600 字 | 600 字⚠️ |
| 📰 News | 700 | ~460 字 | 600 字❌ |

**后果**：故事/文案写到一半被硬截断，没有结尾（用户反馈的核心问题）。

### 1.3 `##标题` 输出污染 TTS

所有 Agent 的 `getOutputFormatRule()` 都要求 LLM 输出 `##标题名称`，但：
- DB prompt 明确说"禁止标题标记(#)"
- TTS 引擎会朗读出 `##` 符号
- 前端虽有 markdown 清洗，但不应依赖前端兜底

### 1.4 前端选项未注入后端

| 模块 | 未传的参数 |
|------|----------|
| 🌙 Radio | `selectedMood.promptHint`（已在之前修复） |
| 📚 Lecture | `selectedLevel.promptHint` |
| 🛒 Ad | `selectedAdStyle.promptHint` + `adProductName` |
| 🎨 PictureBook | `selectedAge.promptHint` |

**后果**：用户选了不同风格/难度/适龄，但 LLM 根本不知道，生成同样的内容。

## 二、解决方案

### 2.1 提示词三层对齐

统一原则：**DB 定角色 + Java 定格式 + 前端定风格，字数三层一致**。

#### DB `ai_role` — 角色定义层

每个模块重写为领域大师级 prompt：

| 模块 | 角色 | 覆盖的品类/主题 |
|------|------|---------------|
| 🌙 情感电台 | 情感文案大师 | 6 种情绪风格 |
| 📚 知识讲解 | 多领域知识大师 | 6 领域（费曼/芒格/卡尼曼/萨根） |
| 🛒 带货文案 | 抖音千万级御用文案师 | 6 品类（数码/美妆/美食/家居/服饰/健康） |
| 🎨 有声绘本 | 养育三个孩子的故事妈妈 | 6 主题（分享/冒险/梦想/环保/情绪/睡前） |
| 📰 新闻播报 | 从事播报三十年的资深主播 | 6 领域（AI科技/航天/财经/国际/气候/民生） |

每个 prompt 统一包含：
1. **铁律**：纯净文本、禁 markdown/emoji、直接进入内容
2. **品类/主题适配**：根据用户选择自动切换风格
3. **心法/配方**：结构化的输出框架（三层讲解/四步转化/魔法配方）

#### Java Agent — 格式控制层

```java
// 修复前
"第一行输出本段标题，格式为 ##标题名称。正文控制在200~350字"

// 修复后
"直接输出正文，不要输出标题行，不要使用任何标题标记。全文控制在600字以内"
```

#### 前端 promptHint — 动态修饰层

```javascript
// 注入到 userInput 里，作为 User Message 的一部分
const levelHint = selectedLevel.value?.promptHint  // "【讲解风格：入门科普】费曼技巧..."
input = levelHint + '\n【硬性要求】全文不得超过600字\n\n' + input
```

### 2.2 maxOutputTokens 统一提升

全部提升到 **1500 tokens**（支持 ~1000 中文字，留足缓冲）：

```java
// 5 个 Agent 统一：
return 1500; // 600字×1.5=900 tokens + 格式缓冲
```

### 2.3 TTS 合成加速

所有单内容类型（radio/lecture/ad/picture_book）统一使用 **100 字分片**：

```javascript
// 修复前：只有 radio 用 100 字分片
const maxLen = project.value?.typeCode === 'radio' ? 100 : 280

// 修复后：4 个单内容类型统一
const maxLen = ['radio','lecture','ad','picture_book'].includes(project.value?.typeCode) ? 100 : 280
```

效果：首片就绪时间从 ~6s 降至 ~2s，用户可以边听边合成。

### 2.4 前端选项注入

```javascript
// generate() 函数中新增：

// 知识讲解
if (project.value?.typeCode === 'lecture') {
  const levelHint = selectedLevel.value?.promptHint || ''
  input = levelHint + '\n【硬性要求】全文不得超过600字\n\n' + input
}

// 带货文案
if (project.value?.typeCode === 'ad') {
  const styleHint = selectedAdStyle.value?.promptHint || ''
  const productHint = adProductName.value?.trim() ? `【产品名称】${adProductName.value}\n` : ''
  input = productHint + styleHint + '\n【硬性要求】全文不得超过600字\n\n' + input
}

// 有声绘本
if (project.value?.typeCode === 'picture_book') {
  const ageHint = selectedAge.value?.promptHint || ''
  input = ageHint + '\n' + input
}
```

## 三、UI/UX 优化

### 3.1 段落卡片标签

单内容类型去掉 "第 X 段/节/页" 的计数标签：

```
修复前: 📚 第 1 节  520字    |  🐰 第 1 页  689字
修复后: 📚 作品内容  520字    |  🐰 作品内容  689字
```

### 3.2 底部发布卡

单内容类型简化统计信息：

```
修复前: 📝 1 段内容  🎧 1 段已配音  📊 520 字
修复后: ✅ 内容已生成  🎧 配音完成  📊 520 字
```

### 3.3 内容展示高度

单内容类型的展开区域从 `max-h-[20vh]` 增大到 `max-h-[50vh]`，避免长文案被截断。

### 3.4 风格选择器

- 带货文案：3 → 4 种风格（新增🔥直播开叫），布局改为 2 列网格
- 所有模块的风格/难度/适龄按钮下方新增 `desc` 描述文字

## 五、面试话术

> **面试官**：你们的 AI 创作模块，提示词是怎么管理的？

### 回答框架

我们设计了一套**三层提示词架构**，解耦了角色定义、格式控制和动态风格：

**第一层：数据库 `ai_role`（可运营热更新）**
- 存在 MySQL 的 `creative_template` 表中
- 定义角色身份、写作哲学、品类/主题适配策略
- 运营同学可以直接改数据库，不用发版

**第二层：Java Agent 代码（编译时固化）**
- `getOutputFormatRule()` — 输出格式规则
- `getTtsAdaptationRule()` — TTS 适配规则
- `getMaxOutputTokens()` — 物理长度上限
- 这些是技术约束，不允许运营随意改

**第三层：前端 `promptHint`（用户实时选择）**
- 难度、风格、适龄等维度的动态修饰符
- 拼接到 User Message 中，随用户选择实时变化

**踩过的坑**：三层之间字数限制冲突。
- DB 说"600 字"，Java 代码说"200 字"，前端说"600 字"
- LLM 收到矛盾指令后行为不稳定
- 解决方案：三层字数统一对齐 + `maxOutputTokens` 作为物理兜底

**关键认知**：
- `maxOutputTokens` 是**硬切**（100% 生效），prompt 里的字数限制是**软控**（~80% 遵守）
- 三层必须方向一致，冲突会让 LLM 混乱
- 前端 markdown 清洗只是兜底，不应依赖它解决格式问题
