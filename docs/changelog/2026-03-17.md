## 🐛 Bug 修复

### Agent 重置对话后记忆仍存在

| 文件 | 改动 |
|------|------|
| `AgentController.java` | `reset()` memoryId 从 `token.hashCode()` 修正为 `user.getId().toString()` |
| `AgentController.java` | 提升 `InMemoryChatMemoryStore` 为共享实例字段 `sharedMemoryStore` |
| `AgentController.java` | `reset()` 调用 `sharedMemoryStore.deleteMessages()` 真正清空记忆 |

**根因**: memoryId 不匹配 + 清错存储位置（清了 Caffeine cache，没清 ChatMemoryStore）

### Agent 流式对话 Tool 调用报错：未找到当前用户上下文

| 文件 | 改动 |
|------|------|
| `SoundReadTools.java` | 移除 `@RequiredArgsConstructor` 和 `ThreadLocal` 显式依赖，增加有状态拷贝构造函数 `SoundReadTools(source, user)` |
| `AgentController.java` | 剥离开销极大的 `OpenAiStreamingChatModel` 为全局单例复用连接池 |
| `AgentController.java` | 接收请求时，动态构建带有 `User` 信息的专属 `SoundReadTools` 实例，以及极其轻量的专属 `StreamingSmartAssistant` 代理 |

**根因与选型**: 
- 原先 `SoundReadTools` 通过 `ThreadLocal` 获取用户上下文，在同步阻塞请求时完全正常。
- 但更换为 `StreamingChatLanguageModel` (SSE) 后，LangChain4j/OkHttp 底层通过**异步线程池**触发 Tool 回调流。此时 Spring 原生的 `ThreadLocal` 已经丢失，引发空指针异常。
- 考虑到高并发安全，**决定不采用任何底层 ThreadLocal Hack**，而是利用代理对象创建开销极低的特性，**将 Agent 架构从“全局单例”解耦为“请求级代理（Request-scoped Proxy）”**，通过重载构造函数完成绝对线程安全的参数级逃逸。

## 🔧 架构调整与重构

### Agent ChatMemory 存储机制重构 (预防 OOM)

| 文件 | 改动 |
|------|------|
| `AgentController.java` | `sharedMemoryStore` 原先被硬编码为 `new InMemoryChatMemoryStore()`（容量无限增长） |
| `AgentController.java` | 重写 `sharedMemoryStore` 匿名内部类，将其底层的存取 `getMessages`/`updateMessages` 对接到受控的 `Caffeine` 缓存上 |

**架构选型与意义**: 
原先的内存记忆无淘汰机制，长时间运行下100%会导致 OOM。通过引入 `Caffeine LRU Cache`，将系统级别的对话记忆容量**强制限制在 500 名活跃用户内**（超载或 30 分钟不发声自动逐出）。这是一个极低成本、抗工业级并发的防护伞，直到未来的多机部署时再平滑替换为 `RedisChatMemoryStore`。

### ReAct 架构与 Prompt 产品级重写

| 文件 | 改动 |
|------|------|
| `SmartAssistant.java`<br>`StreamingSmartAssistant.java` | 抛弃早期单纯的角色人设，重写为**“思考 (Thought) → 行动 (Action)”**的产品级 ReAct 架构 |

**意义**: 原有的 Prompt 倾向于和用户“闲聊”确认需求，导致行动迟缓。新架构将用户意图切分为 A/B/C/D 四个清晰等级，设定“红线规则”与“默认参数”，极大增强了 AI 的行动力和出音效率。
